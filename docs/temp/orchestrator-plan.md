# План доработки: роль-оркестратор и JSON-пейлоад

## Цели
- Добавить специальную роль-оркестратор, которая может обрабатывать все сообщения группы.
- Дать возможность включать/выключать роль и переключать режим оркестратора.
- Перейти на структурированный формат входа в LLM: данные собираются как JSON-объект.
- Важно: в провайдер JSON уходит как текстовая строка (serialized JSON), а не как отдельный JSON transport-контракт.

## Важное уточнение по формату (обязательно)
- Мы формируем структуру `dict` в коде.
- Перед отправкой сериализуем в строку (`json.dumps(..., ensure_ascii=False)`).
- В `LLMRouter.send_message(...)` и шаблонах провайдера это значение передается как обычный `content` (text).
- Для LLM это текст с JSON-содержимым.
- Транспортный контракт провайдера не меняем: отправляем как и раньше текстовое поле.
- Схема JSON должна соответствовать документу `docs/temp/orchestrator-json-schema.md`.

Это корректно и совместимо с текущей архитектурой, если:
1. Модель/провайдер нормально переваривает длинный JSON-текст.
2. Размер пейлоада не превышает лимиты провайдера.
3. Схема JSON стабильна и версионируется.

## Итерация 1

### Текущий статус (на 2026-02-21)
- Шаги 1-11 выполнены в коде.
- Следующий системный этап: унификация flow через единый engine (см. `docs/temp/orchestrator-engine-plan.md`).
- Реализованы:
  - единый executor отправки роли (`_execute_role_request`);
  - post-response в orchestrator после ответа роли;
  - dispatcher делегаций role->role по тегам `@role` в ответах ролей/оркестратора;
  - антицикл: лимит одинаковых делегаций по тексту (3 повтора) + аварийный лимит шагов цепочки `orchestrator_max_chain_auto_steps`;
  - chain-logging (`chain_id`) для трассировки делегаций.
- Остаток: ручной Telegram smoke по чеклисту из `docs/temp/step7-regression-report.md`.

### 1. Модель данных и миграции
- Расширить `group_roles`:
  - `enabled INTEGER NOT NULL DEFAULT 1`
  - `mode TEXT NOT NULL DEFAULT 'normal'` (`normal|orchestrator`)
- Обновить `GroupRole` в `app/models.py`.
- Добавить методы в `Storage`:
  - `set_group_role_enabled(...)`
  - `set_group_role_mode(...)`
  - `list_enabled_roles_for_group(...)`
  - `get_enabled_orchestrator_for_group(...)`
- Ограничение: только один `enabled orchestrator` на группу.

### 2. Управление через Telegram UI
- В `callbacks.py` добавить действия:
  - включить/выключить роль;
  - назначить роль оркестратором;
  - снять режим оркестратора.
- Отображать текущий статус роли в карточке (enabled/mode).

### 3. Роутинг сообщений группы
- В `messages_group.py`:
  - если в группе есть enabled оркестратор и в сообщении есть `@role`/`@all`, сообщение идет в соответствующие normal-роли;
  - после ответа normal-роли бот отправляет post-response событие в оркестратор;
  - после ответа роли/оркестратора из текста ответа извлекаются `@role` и выполняется внутренняя делегация в целевые роли;
  - если role-mention отсутствует, сообщение идет напрямую в оркестратор;
  - добавить защиту от петель: игнорировать сообщения бота.
- Опционально сохранить текущий `owner_user_id` фильтр (как сейчас), чтобы не менять security-модель.

### 4. JSON payload builder (как текст)
- Добавить новый builder в `app/services/prompt_builder.py`, например:
  - `build_llm_payload_json_text(...) -> str`
- Включить поля:
  - `actor` (`username` с правилом источника: user / role / orchestrator)
  - `instruction` (`system`, `message`, `reply`)
  - `context.reply` (`is_reply`, `previous_message`)
  - `user_request` (`text`, `recipient`)
  - `llm_answer` (`text`, `role_name`) для post-response в оркестратор
  - `tools.available`
- Весь объект сериализовать в строку JSON.

### 5. Отправка в провайдер (без ломки контрактов)
- `LLMRouter.send_message` оставить принимающим строку для `content`.
- На уровне handlers использовать новый JSON-text builder вместо старой склейки строк.
- Для провайдеров менять transport не нужно: `{{content}}` остается строкой.
- Рекомендуемый формат текста для модели: префикс `INPUT_JSON:` + JSON-строка (или fenced block `json`).

### 5.1. Парсинг ответа оркестратора
- Для роли `mode=orchestrator`:
  - сначала пытаться распарсить ответ как JSON по схеме ответа;
  - если JSON невалиден/неполон — fallback к обычному текстовому ответу (без падения обработки).
- Для `normal` ролей поведение остается текущим (текстовый ответ).

### 6. Логирование и наблюдаемость
- Логировать:
  - что роль работает как orchestrator;
  - размер JSON-text payload;
  - смену `session_id` при recovery;
  - включение/выключение ролей.
  - делегации: `detected/sent/skip/failed` + `chain_id`.

### 7. Регрессия
- Проверить:
  - normal роль по `@mention`;
  - orchestrator на всех сообщениях;
  - toggle enabled/mode;
  - отсутствие петель;
  - корректный fallback при невалидном JSON-ответе оркестратора;
  - корректный fallback при ошибках LLM.

## Потенциальные проблемы и как снизить риск
- Слишком большой JSON-text: ограничение размера + сокращение полей.
- Нестабильный формат: фиксированный ключевой набор полей и версионирование в документации.
- Смешение orchestration и обычных ролей: строгая проверка `mode/enabled`.
- Конкуренция правок ролей вручную и оркестратором: блокировка/последняя запись + аудит в логах.
